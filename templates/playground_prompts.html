📖 How memory works (two-bucket system) — click to read

Maya's memory of each user is split into two buckets with different lifecycles. This split is the foundation of how personalization works in this app.

Bucket 1 — Facts

Timeless truths about the user. Never expire. Always loaded into the prompt. Updated only when the user explicitly contradicts. Stored as a key→value object.

Examples: name, profession, mother tongue, family, hometown, broad interests, past life experiences ("broke leg playing cricket at 16"), english_profile ("nervous about speaking").

Bucket 2 — Events

Things tied to a specific calendar date. Auto-archived 14 days after the date passes. Always have a date field. Stored as an ordered list with id, what, date, added, followed_up.

Examples: GMAT exam on May 15, sister's wedding May 22, got promoted last Friday, doctor appointment tomorrow.

How it's stored on disk

One JSON file per user at memory_store/{user}.json. Sample contents: 

{
  "facts": {
    "name": "Sunil",
    "profession": "Hardware sales at a small firm in Indore",
    "mother_tongue": "Hindi",
    "location": "Indore",
    "family": "Lives with parents and younger sister",
    "interests": ["cricket", "Bollywood films", "biryani"],
    "background": [
      "Played cricket for school team in Indore",
      "Father worked at LIC for 30 years"
    ],
    "english_profile": "Intermediate. Freezes in client meetings. Drops articles."
  },
  "events": [
    {
      "id": "ev_001",
      "what": "Client call with US customer about hardware order",
      "date": "2026-04-30",
      "added": "2026-04-28",
      "followed_up": false
    },
    {
      "id": "ev_002",
      "what": "Got first English-medium client account",
      "date": "2026-04-21",
      "added": "2026-04-21",
      "followed_up": true
    }
  ]
}

How it's rendered into the prompt Maya sees

The structured JSON above gets formatted into a natural-language block before being injected into Maya's first-message instruction. Dates are translated to relative language client-side so Maya can speak naturally: 

About you (always true — never expires):
  • Name: Sunil
  • Profession: Hardware sales at a small firm in Indore
  • Mother Tongue: Hindi
  • Location: Indore
  • Family: Lives with parents and younger sister
  • Interests: cricket, Bollywood films, biryani
  • Background: Played cricket for school team in Indore, Father worked at LIC for 30 years
  • English Profile: Intermediate. Freezes in client meetings. Drops articles.

Coming up / recent events (date-bound, will auto-archive):
  • 7 days ago (April 21): Got first English-medium client account (id: ev_002)  ← already followed up, do not re-raise
  • in 2 days (April 30): Client call with US customer about hardware order (id: ev_001)

How memory gets updated each session

At the end of each chat (/end_session), one LLM call runs the memory_merge prompt. It takes the current facts + events + the new transcript and returns a structured JSON patch. The server applies the patch deterministically — no risk of accidentally losing your name during a chatty session.

Sample patch the merge LLM might emit (after Sunil mentions his client call went well and he's now planning to attend a sales conference in Mumbai on June 10):

{
  "facts_appends": {
    "background": ["Closed first international hardware deal April 30"]
  },
  "events_add": [
    {"what": "Sales conference in Mumbai", "date": "2026-06-10"}
  ],
  "events_followed_up": ["ev_001"]
}

The server applies this: appends to background, adds the new event with auto-generated id, marks ev_001 as followed up, and auto-archives anything older than 14 days past its date.

Sample memory_merge prompt — for reference only

Below is an illustrative version of the merge prompt with a worked example. This is NOT the prompt the app actually sends — the live one is in the memory_merge card below. This sample is here so you can see the shape of what the LLM is asked to do. Use it as inspiration if you're customizing the live prompt. 

You are updating a structured memory store with two buckets:
  FACTS  = timeless truths about the user (no date)
  EVENTS = things tied to a calendar date (auto-expire after 14 days past)

TODAY'S DATE: 2026-04-28

CURRENT FACTS:
{
  "name": "Sunil",
  "profession": "Hardware sales at a small firm in Indore",
  "english_profile": "Intermediate. Freezes in client meetings."
}

CURRENT EVENTS:
[
  {"id": "ev_001", "what": "Client call with US customer", "date": "2026-04-30",
   "followed_up": false}
]

NEW TRANSCRIPT (most relevant excerpts):
User: Hi Maya! The client call I told you about went really well today.
User: They confirmed the order. Big win for me.
User: Also I have to attend a sales conference in Mumbai on June 10.
Maya: Oh wonderful! How exciting. Tell me about Mumbai.

EXPECTED PATCH OUTPUT:
{
  "facts_appends": {
    "background": ["Closed first international hardware deal April 30"]
  },
  "events_add": [
    {"what": "Sales conference in Mumbai", "date": "2026-06-10"}
  ],
  "events_followed_up": ["ev_001"]
}

NOTE what we did NOT include:
- Maya asked about Mumbai, but the user did not say anything substantive about it,
  so nothing about Mumbai goes into facts.
- The "April 30" outcome got promoted into facts.background (timeless achievement)
  AND ev_001 was marked followed_up (so Maya stops asking about it).

Lifecycle rules the live merge enforces

  • Fact fidelity: only persist what the USER stated, never what Maya asked or invented.
  • Date resolution: "tomorrow" / "next week" / "yesterday" are converted to absolute YYYY-MM-DD before persisting.
  • No fact churn: only update a fact field when the user contradicts or specifies more precisely.
  • Auto-archive: events whose date is more than 14 days past TODAY get dropped on the next save.
  • Promotion: when an event resolves into a timeless achievement, the merge can append it to a facts list (e.g. background).
