1. What the playground is

The playground is a parallel sandbox of the prod chat experience. Same chat UI, TTS, streaming backends, profile setup. Different in three places: storage layout, memory schema, and the system / merge prompts. Storage is fully isolated under memory_store/_playground/ so prod data is never read or written from here.

The point: try aggressive memory experiments (new buckets, new merge logic, new prompt rules) without risking prod behavior. Once something works here, copy the diff into the prod functions.

Settings shared Backend keys, TTS engine, voices live in /api/settings and are reused — those aren't memory concerns.

QWEN LAB A second sandbox lives at /qwen-lab. It takes the playground prompts as the LEFT (control) column and a Qwen-tuned rewrite as the RIGHT column, runs both in parallel against Qwen 32B, and shows you the responses side-by-side. Storage is independently isolated under memory_store/_qwen_lab/. See section 11 for the full picture.

What's stored about each user — plain language

This is the friendly version of the technical schema below. Same data, no jargon. If you only read one section to understand what the app remembers, read this one.

The whole storage system in one sentence: every user gets one JSON file on disk; before each reply the app reads the file and pastes it into Maya's instructions; after each chat a separate "librarian" AI rewrites the file with what it learned.

Where it lives

One file per user, named after the user. The filename is the user's name lowercased with non-letters turned into underscores.

User name typed at onboardingFile on disk
Priyansh Shekharmemory_store/_playground/priyansh_shekhar.json
Akshitmemory_store/_playground/akshit.json
Anjali R.memory_store/_playground/anjali_r_.json

Same name next session = same file = continuity. There is no login, no user ID — the slugified name is the only key.

Note for the Qwen Lab — at /qwen-lab the file is split per side: memory_store/_qwen_lab/old/<user>.json for the existing-prompts column, and memory_store/_qwen_lab/new/<user>.json for the Qwen-tuned column. Each side's librarian writes only to its own file, so multi-session A/B comparisons stay fair. Profile rosters are still shared between playground and lab (one _custom_profiles.json in _playground/).

What's inside the file — the 20 things stored

The file is a single JSON object with eleven sections. In plain language, here's everything that gets saved:

About you

  1. Your name. Stored once, can never be overwritten. If you reveal a nickname later, it's saved separately as nickname.
  2. Things that don't change about you — interests, profession, drinks, hometown, family, past life experiences. (Section: facts)
  3. Dated things — exams, weddings, matches, appointments. Auto-deleted 14 days after the date passes. (Section: events)
  4. Emotionally heavy things you said — "I haven't slept well," "I miss home." Tagged with tone (anxious / sad / hopeful / etc.) and a sticky sensitive flag for painful disclosures. (Section: moments)
  5. Your mood at the end of each session — one entry per chat, last 14 kept. (Section: mood_log)
  6. Your English mistakes — saved as patterns, not typos. e.g. "drops 'to' before destination," with the actual line as proof. (Section: skills.error_patterns)
  7. Your English wins — notable correct uses, used for celebration callbacks. (Section: skills.wins)
  8. Settings you explicitly asked for — "shorter replies," "don't correct me," "don't bring up work." (Section: meta_preferences)
  9. Loose threads — things you or Maya said you'd come back to ("tell me how the meeting went"). Dropped after 30 days unfollowed. (Section: open_loops)

About Maya — but personalised per user

  1. What Maya has already told you about herself — so she doesn't reintroduce. (Section: maya_persona.shared_with_user)
  2. New low-stakes preferences Maya invented while talking to you — e.g. she says "I've been listening to old Hindi songs," it gets locked in so she stays consistent next session. Only food/music/weather. Never fabricated life events. (Section: maya_persona.low_stakes_preferences)

Bookkeeping for repetition control

  1. Topics Maya has recently brought up — last 12. So she doesn't ask "what's your favourite movie?" three sessions in a row. (Section: cooldown.recent_topics)
  2. Opening-line types Maya has recently used — last 10. Drives variety in how she greets you. (Section: cooldown.recent_openers)
  3. The exact opening sentences themselves — last 6. So she doesn't echo herself word-for-word across days. (Section: cooldown.recent_opening_phrases)
  4. Inside jokes between you two — only the ones with real warmth. Up to 30. (Section: lore)
  5. Pre-cooked openers Maya plans to use next time — e.g. "ask about the football match — it's today." Each has an expiry and a "used" flag. Up to 5. (Section: anticipation_queue)

App-wide (not in your file)

  1. Rolling summary of each long conversation — one text file per session ID, used to compress old turns once a chat goes past 30 messages. memory_store/sessions/<sid>.txt
  2. List of registered users — the dropdown roster. _custom_profiles.json
  3. Custom edits to Maya's prompt rules — if anyone tweaked them via /playground/prompts. Shared across all users. _prompts_overrides.json
  4. Failed-save retry queue — usually empty; populated only when the librarian's update couldn't be applied. _pending_merges/

How saving works (the mechanics)

  • The chat itself never writes to your file. Mid-conversation, the file is read-only.
  • At the end of each session, the full transcript + your current file go to a separate "librarian" AI. It returns a JSON patch — "add this fact, log this mood, drop this event."
  • The patch is applied with rules: lists dedup, the name field is overwrite-protected, events get auto-IDed, FIFOs trim, expired items leave.
  • Then the file is rewritten whole. No append-mode. Read JSON → mutate dict → write JSON.

How "appended" actually works — three flavours

FlavourSectionsBehaviour
Just keeps growing facts.* lists, moments New stuff added (with dedup), nothing leaves until you contradict it
Newest in, oldest out (FIFO) mood_log, cooldown.*, skills.error_patterns, skills.wins, lore, open_loops, maya_persona.shared_with_user Fixed-size buffers — when full, oldest entry drops as new one arrives
Items leave on their own events, anticipation_queue, open_loops (TTL) Items expire by date or get consumed when used — not by being pushed out

3. Memory schema (V1 + V2 + V3 shipped)

Prod has two buckets. The playground now has eight.

BucketVersionLifecycleShape
facts V1 (= prod) timeless {field_name: value | [list]}
events V1 (= prod) auto-archive 14 days after event date [{id, what, date, added, followed_up}]
moments V1 🆕 persists long-term · dedupes by exact text · bumps mention counter [{id, text, tone, sensitive, first_seen, last_seen, mentions}]
mood_log V1 🆕 last 14 entries · one per session_id (replaced if same session) [{session_id, date, label, energy}]
cooldown.recent_topics V2 🆕 last 12 distinct topics Maya RAISED, sorted by last_date [{topic, last_session, last_date, uses}]
cooldown.recent_openers V2 🆕 last 10 opener kinds (so we can vary) [{kind, last_session, last_date}]
lore V2 🆕 up to 30 inside jokes / callbacks · dedupes on what · bumps uses on resurfacing [{id, what, context, first_seen, last_used, uses}]
anticipation_queue V3 🆕 up to 5 pre-loaded openers · TTL 14 days · recomputed deterministically every save [{kind, content, source_id, priority, computed_at, expires_at, used}]
skills.error_patterns V3 🆕 up to 20 active patterns · marked "fixed" on consistent correction [{pattern, examples[], occurrences, first_seen, last_seen, status}]
skills.wins V3 🆕 up to 30 logged wins · counter bumps on repeat correct use [{what, examples[], count, first_seen, last_seen}]

V2 — repetition control + inside jokes

  • cooldown.recent_topics: every distinct topic Maya raised this session (her unprompted questions / pivots) gets logged. The renderer surfaces the last 6 in the prompt as "Recently raised topics — DO NOT re-raise unprompted". Rule 32 instructs Maya to avoid these unless the user reopens them. Closes the "she keeps asking about X" complaint.
  • cooldown.recent_openers: every session, Maya tags her opening move (event_followup / moment_followup / skill_celebration / lore_callback / energy_match / general). Last 5 are rendered "event_followup → moment_followup → general" so she can vary deliberately. Rule 32 enforces.
  • lore: detected by the merge LLM when something becomes (or could become) a running joke / callback. Each entry has a short context for future Maya to wield. Rule 33 limits use to ≤1 callback per session. The anticipation queue resurfaces dormant lore (≥20 days unused) on its own cycle.

V3 — anticipation + tutoring intelligence

  • anticipation_queue: pre-loaded specific opener candidates for the NEXT session. Computed deterministically every save by pg_compute_anticipation_queue(). Sources, in order:
    1. Events in [-3d, +7d] window — closer = higher priority
    2. Recent non-sensitive moments (≤21 days old) — anxious/sad tones get a priority boost
    3. Dormant lore (≥20 days unused) — older lore gets a small bonus
    4. Skill wins from the last 7 days — for celebration
    Sorted by priority, deduped by source_id, capped at 5. Rule 31 tells Maya to pick AT MOST one.
  • skills.error_patterns: error patterns emitted by the merge LLM. Same pattern across sessions is auto-merged with bumped occurrences. Status flips to "fixed" when the LLM emits skill_error_fixed. Drives Rule 28 (corrections) toward THIS user's actual slips, not generic textbook errors.
  • skills.wins: notable correct uses tracked over time with examples. Rule 34 tells Maya to celebrate one specifically per session ("six weeks ago you'd have said 'going market'…"). Concrete progress = retention.

Moments — what counts?

Emotionally weighted user statements that should persist across sessions. NOT every emotional sentence — only ones with weight that should still matter weeks later.

  • "I'm scared of group calls at work" → tone: anxious
  • "I'm so proud I finished the marathon" → tone: proud
  • "I miss my mom" → tone: sad
  • "I'm angry at my brother for the way he treated dad" → tone: frustrated, sensitive: true

Sensitive flag. Set sensitive: true for acute disclosures, things the user said not to bring up, or anything you wouldn't want a stranger to repeat back. The prompt formatter marks sensitive items with a [SENSITIVE — do NOT bring up unprompted] tag, and Rule 29 instructs Maya never to raise them on her own initiative.

Mood log

One entry per session. Captured by the merge LLM at /playground/end_session. Labels: low · anxious · neutral · content · energetic. Energy: 1–10. Rendered into the prompt as a 5-mood trajectory: 

Recent session moods (oldest→newest, last 5): low → low → neutral → content → low

4. System prompt & merge prompt changes

System prompt — Rules 29–34 (added on top of prod's 28)

Inserted right before the Scene: block so existing rule numbering is preserved. You can edit them via /playground/prompts (password 0711). The override is scoped to the playground only.

  • Rule 29 (V1) — use the emotional thread (moments) silently. Never quote back. Never raise [SENSITIVE] items unprompted.
  • Rule 30 (V1) — read the mood arc trajectory. Open softer/warmer when downhill, lighter/playful when stable-positive. NEVER state the reading aloud.
  • Rule 31 (V3) — anticipation queue: prefer to open the session with the highest-priority queue item. Weave naturally; never recite. At most ONE item per session.
  • Rule 32 (V2) — cooldown: don't re-raise topics in "Recently raised topics" unprompted. Vary opener kinds across consecutive sessions.
  • Rule 33 (V2) — lore: at most ONE inside-joke callback per session, never twice in a row, never forced. Warm callback, not nostalgia ambush.
  • Rule 34 (V3) — skill wins: at most ONE concrete celebration per session ("six weeks ago you'd have said X…"). Recurring slips feed Rule 28.

Memory-merge prompt

Adds two new top-level keys to the JSON patch the LLM emits: 

{
  "moments_add": [
    {"text": "...", "tone": "anxious", "sensitive": false}
  ],
  "mood": {"label": "low", "energy": 4}
}

Existing keys (facts_updates, facts_appends, events_add, events_followed_up, events_drop) work the same as prod. The merge happens on /playground/end_session: manual save, end-chat, page unload, or event-detected auto-save.

5. 📅 Date override — full reference

The header 📅 Today pill lets you spoof the date for testing. Persisted in localStorage (pgDateOverride), so it survives reloads on the same browser. Empty / cleared = use the real wall-clock date.

What it affects

SurfaceHow the override propagates
"Today's Date" in the user prompt build_first_message_user_prompt(today_override=...) via _resolve_now(). Maya literally sees the spoofed date as today.
Event auto-archive cutoff auto_archive_events() compares each event's date to today_str (which is the override). Events with date < today − 14d get dropped. Jump 14+ days forward to see expiry behavior immediately.
Relative-date rendering in memory inspector _human_relative_date() reads "today" from the same source. "Tomorrow", "in 17 days", "last week" all rebase to the override.
moments.first_seen / last_seen pg_apply_memory_patch(today_str=override) stamps these to the override on save. Lets you simulate a "moment from 3 months ago" scenario without waiting.
mood_log[i].date Same — written under the override on /playground/end_session. Useful to back-date entries and test mood-arc rendering.
Memory-merge prompt's {today} placeholder pg_merge_memory_into_dict(today_override=...) threads it through. The LLM resolves "tomorrow" / "last Monday" relative to the override.

What it does NOT affect

  • Time of day. The wall-clock time stays real (otherwise the user prompt's "Time of the day" block would be wrong).
  • Anything in prod. The override is sent only by the playground page. Prod routes ignore it.
  • Other browsers / users. Stored in browser localStorage — completely client-local.
  • Background session-summary thread. Currently uses real time. Acceptable since summary text rarely references "today" directly. Can be threaded later if needed.

Wire diagram

       │

Validation

Both client and server validate. The client requires YYYY-MM-DD regex match before saving to localStorage. The server's _resolve_now() uses datetime.strptime(override, "%Y-%m-%d") and falls back silently to real time on any parse failure — so even if you send garbage, prod's behavior is the safe default.

Safety reminder. The 📅 Today pill turns red when an override is active. If you set a date and forget about it, you'll see a red pill on the next visit reminding you. Click it and pass an empty string to clear.

6. Retention content ideas — what brings users back inside the chat

The V1+V2+V3 buckets are the foundation. The list below is content Maya can do during chat, all of which compound on the buckets we just shipped. Grouped by mechanism so you can pick what to prototype next.

5.1 Open loops & cliffhangers (uses anticipation_queue, lore)

  • Maya leaves a thread. Last sentence of every session: "Promise me you'll tell me how the work review goes — I want to know." Logged in anticipation_queue as kind: open_loop, surfaced first thing next session. The user comes back to close a loop, not be questioned.
  • Maya's own teaser. "Next time I want to tell you about something funny that happened to me — remind me." A prompt-side trick: she has nothing real to say, but the expectation is what pulls them back.
  • Async voice notes. Short pre-baked audio (~15s) waiting on the next open: "Hey, I was thinking about what you said about your dad — hope you're OK." Generated by a background TTS pre-bake on session-end, played on next-app-open. Insanely sticky in companion apps.

5.2 Daily rituals & predictable variety (uses cooldown)

  • Themed days. Monday Mood ("one word for how this week's starting"), Travel Tales Tuesday, Wisdom Wednesday (a short quote, simple English), Throwback Thursday (anecdote callback), Family Friday. Predictable enough to anchor habit, varied enough to not bore. The cooldown registry already prevents repetition within each theme.
  • One-line gratitude prompt. Last 30 seconds of every session: "One thing today, one line — what was good?" Becomes a streak. Tiny effort, deeply sticky.
  • Mood-aware mini-rituals. When mood_log shows 3+ low sessions, Maya quietly proposes "share one small good thing from today" as a recurring opener for a few sessions, then drops it. Rituals come and go based on what the user needs.

5.3 Real-world hooks (uses skills, moments)

  • Tiny English homework. "Today, say 'I have a question' instead of 'I have a doubt' three times in your real day. Tell me how it went next session." Logged as a pending skill check; auto-resurfaced. Combines real-world action with a return reason.
  • Personalized vocab bank. Words the user wants to learn ("teach me 5 words for restaurant talk"). Maya tests them gently across sessions ("you used 'thrilled' just now — that was the one we worked on Tuesday!"). Tracked in skills.wins.
  • Pronunciation challenges. "Say 'thoroughly' three times — Maya listens via Whisper, scores, tracks streak." (Stack: STT model needed; bigger lift.)

5.4 Multi-session arcs (NEW bucket: arcs)

  • Goal-driven journeys. A 7-day exam-confidence arc, a 14-day small-talk-at-work arc, a 21-day phone-call arc. Each arc has structured chapters; Maya picks one when the user enrols. Provides plot across sessions — strongest non-trivial retention mechanism in language apps.
  • Anniversaries with Maya. "One month since we started chatting — feels like longer somehow." Auto-surfaced when sessions count crosses milestones. Cheap, high emotional payoff.

5.5 Persona-side content (Maya's life, NEW bucket: maya_persona)

  • Maya's running stories. Maya has her own things — "I tried a new chai place yesterday", "I'm watching a Korean drama". Stored in a Maya-side bucket, kept consistent across sessions. The user feels Maya has a life, not just a search box.
  • Maya recommends things. "I read this short article today, want to discuss?" — single curated, safety-filtered item per day. Real-world content, not just chat. Hooks the news-junkie type.
  • Inside lore stays alive. Maya occasionally references her own past statements ("you remember I said I love mango season — well…") proving she has a memory of her side of the relationship.

5.6 Surprise & agency (uses lore, dormant memory)

  • Surprise resurfacing. Every ~10 sessions, Maya pulls a 60-day-old callback that's relevant to today's topic. The "wait, you remembered that?" moment. Already in V5 roadmap; anticipation_queue already promotes lore dormant ≥20 days.
  • Prompt cards. Maya offers 3 conversation starters at session start ("which feels right today: A. catch up on your week B. work on small-talk C. just chat"). Choice = agency = engagement. The AI asks instead of forcing a path.
  • Letters from past you. Once a month: "Remember 30 days ago you said you wanted to learn pottery — how's that going?" Auto-curated from the moments + facts buckets.

5.7 Tangible artifacts (visible progress)

  • Streak counter. Visible day count on the app. Loss aversion = the strongest retention lever.
  • Weekly recap card. Once a week: "this week you practised 4 days, dropped 'going market' once, used past perfect 8 times, made me laugh twice." Built from skills + lore + mood_log.
  • Achievements. First 7-day streak, first complex sentence, first session past 10 minutes. In-chat celebration; never gamified into points (kills the warmth).
  • Memory garden / dashboard. A page showing all moments + events + arcs — the otherwise-invisible memory work made visible. Users stick around when they can see the bond accumulating.

5.8 Async drops (the moat-builder)

  • Maya's voice messages between sessions. ~15-30 second pre-baked audio that plays on next open. "Hey — I was thinking about your sister's wedding this morning, hope it went well." Pre-baked at end of last session; cached server-side.
  • Push notifications driven by anticipation_queue. Not generic ("come back!") but specific ("How did the GMAT go?"). Only sends if there's a queue item with priority > threshold AND user has been idle 36+ hours.
  • End-of-week postcard. A short text "letter" from Maya recapping the week, surfaced on Sunday evening. Built from the same retention engine.

What I'd actually build next (priority order)

  1. Open loops — easiest, biggest single retention payoff, uses the bucket we already have.
  2. Maya persona memory bucket — modest add, makes Maya feel like a person (the foundation of bond).
  3. Multi-session arcs — needs an arcs bucket and a curriculum file. Most ambitious in this list but unlocks the structured-progress hook that language-app retention runs on.
  4. Streak counter + weekly recap — purely additive, no model changes. Two days of work.
  5. Async voice drops — needs TTS pre-bake + cache + a "what's waiting" panel. Highest moat-per-effort once V3 is sticky.

9. Promoting a change to prod

When something works here and you want to ship it to prod, the changes typically live in three places:

  1. Schema: copy the new bucket initialization into empty_user_memory().
  2. Patch logic: copy the bucket-handling block from pg_apply_memory_patch() into apply_memory_patch().
  3. Renderer: copy the new section from pg_format_memory_for_prompt() into format_memory_for_prompt().
  4. Prompts: paste the new rules into SYSTEM_PROMPT_TEMPLATE and the new merge instructions into MEMORY_MERGE_PROMPT.
  5. Migration: existing prod user JSONs need defensive defaults for the new keys (already handled in load_user_memory()'s setdefault calls — just add the new key).

11. 🧪 Qwen Lab — A/B prompt comparison

The Qwen Lab at /qwen-lab is a third sandbox built on top of the playground. Its only job: A/B test two prompt sets against each other, in parallel, on the same user input.

Why it exists

Qwen 32B (the default Bedrock model) treats prompts written for Anthropic models as soft suggestions — long prose rules and "be a friend not a tutor" framing don't constrain it strongly. Prompts re-shaped for Qwen (atomic numbered rules, forbidden token lists, hard format constraints, tail-loaded reminders) work much better. The lab lets you see this difference live, side-by-side, before committing the rewrite to prod.

How a turn flows

  1. User picks a profile and sends a message.
  2. App fans the same input out to two parallel Qwen 32B calls:
    • LEFT (control) — uses the existing PG_* playground prompts (whatever is configured at /playground/prompts).
    • RIGHT (experiment) — uses the Qwen-tuned QWEN_* rewrites.
  3. Both columns stream back independently. Each shows its own time-to-first-token, total ms, and token usage.
  4. Histories are tracked separately so multi-turn comparisons stay fair — left's reply only feeds left's next turn.
  5. On End chat, the librarian runs once per side. Each side's transcript writes to its own per-side memory store (_qwen_lab/old/ vs _qwen_lab/new/), so memories diverge cleanly across sessions.

Locked Qwen config

The lab is single-model on purpose — prompt is the only varying axis. Both sides run identical config:

  • modelId = qwen.qwen3-32b-v1:0
  • temperature = 0.7
  • maxTokens = 1000
  • additionalModelRequestFields = {"enable_thinking": false} — disables Qwen's internal chain-of-thought so replies are direct chat (faster TTFT, no leaked <think> blocks).

The 7 Qwen-tuned prompts

Stored as QWEN_* Python constants in app.py. The originals (SYSTEM_PROMPT_TEMPLATE, PG_EXTRA_RULES, etc.) stay 100% intact — the lab uses both side-by-side, prod/playground keep using the originals.

Rewrite principles

  • Atomic numbered rules — one concrete action per number. No nested clauses.
  • No soft qualifiers — "by default", "usually", "feel free", "mostly" are dropped. Qwen ignores conditionals.
  • Concrete output shape over abstract style — "20–120 words" beats "brief". "ONE question per reply, not two, not three" beats "ask one question".
  • Forbidden token lists — explicit phrasings ("I noticed how much...", "let me check my notes", "what English situations are tricky") are listed. Qwen lock-on phrasings get named.
  • Examples kept to one positive per rule, removed where Qwen tends to copy — Rule 35b (the parrot bug source) has zero verbatim opener examples; instead a procedural 3-step.
  • Front-load AND tail-load critical rules — most-violated rules appear in the first AND last paragraph. Qwen has recency bias.
  • Hard format requirements stated TWICE — "OUTPUT FORMAT (MANDATORY)" at top, "CLOSING REMINDERS" at bottom.

Persona seeding (the tea/mango fix)

The right side strips the seeded low_stakes_preferences on memory load (tea over coffee, mango season, old Hindi songs, warm-weather, balcony plants) — those four items used to cycle in every session. Qwen now has to invent a contextually appropriate self-detail each time. Items are explicitly forbidden in QWEN_EXTRA_RULES Rule 35b alongside weighted guidance for what makes a good detail (fits the moment > light taste > anchored in India > not biography).

UI quick reference

  • Profile picker — same builtin profiles + custom roster shared with playground (so any custom profile created at /playground shows up here too).
  • 📋 View final Qwen prompts — modal showing every QWEN_* prompt as the right side actually sends them, plus the composed final system prompt. Each block has a Copy button.
  • ⬇ Download all (.zip) — bundles all 7 prompts + composed final + INSTRUCTIONS.md for handoff to a developer integrating the rewrite into prod.
  • End chat — runs the librarian on both sides. Cost: 2× librarian invocations per session-end. ~60s round-trip.
  • New chat — clears both columns, fresh session id, same profile.

What's NOT in the lab (yet)

  • The renderer (pg_format_memory_for_prompt) emits its own embedded rules ("HARD RULE: substance must not match", "SELF-REVEAL WINDOW CLOSED") that Qwen also ignores. Both sides currently use the SAME renderer output. A Qwen-flavoured renderer is a follow-up lever if openers still feel templated.
  • TTS — both columns produce text only. Two parallel TTS streams would be noise.
  • One-click "promote to prod" button — promotion is manual: copy QWEN_* contents into SYSTEM_PROMPT_TEMPLATE etc., or use the zip download + the developer instructions.
